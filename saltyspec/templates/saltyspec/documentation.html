How It Works

SaltySpec uses SALT GUI to generate Use Case Specification. In short, this is what the generation process looks like:

You might be wondering how SALT GUI can be translated to Use Case Specification. Well, we have modeled a procedure in order for this to work.

Output

First, let’s talk about the output as it is our main objective to reach. Understanding the output will help to visualize the requirements or variables needed in order to achieve the desired output. Use Case Specification is . In general, a use case specification would contain following items:

  1. Title
  2. Description
  3. Actors
  4. Pre-condition
  5. End-condition
  6. Main scenario

Input

Second, from what we know of the output, we have to take a look at the input. SALT GUI is . SALT GUI is basically a code which represents GUI (mostly in the form of input form). In order to use the application, we have to understand how to write SALT GUI. You can learn more about it here.

Now, to understand how things work in SaltySpec, here is the working example of SALT GUI that can be translated to Use Case Specification with SaltySpec: 


@startuml

title Login     

!unquoted procedure SALT($x)
"{{
salt
%invoke_procedure("_"+$x)
}}" as $x
!endprocedure

!procedure _form()
{+
<b>Login
username
"username"
password
"********"
[login]
Don't have an account?
__create an account__
}
!endprocedure

!procedure _success()
{+
<b>success
congratulations!
[ok]
}
!endprocedure

!procedure _error()
{+
<b>error
try again
[ok]
}
!endprocedure

(*) --> SALT(form)
form --> SALT(success)
form --> SALT(error)

@enduml

Those code will produce something like this:

Notice that there are parts in the example given. Following are the parts that need to be provided in order to use the application:

  1. Title, is part that’s going to be used as the title of the Use Case Specification.
  2. !procedure _form(), is the part that represents the form or GUI that is being used.
  3. !procedure _success(), is the part that’s going to be used as the main scenario. It should contain what the desired output looks like.
  4. !procedure _error(), is the part that’s going to be used as an alternative scenario. It should contain what the alternative output looks like.

Those are the parts that SHOULD be provided in the input form of this application.

How to Use

There are a few steps that must be done to convert the SALT GUI into Use Case Specification. Here are the following steps:

  1. Input the actor of the use case in the “Aktor” field.

  1. Input the Salt GUI code in the “Salt” field (with the specified criteria).

  1. Make sure the Salt GUI has all of the specified requirements / criteria. 
    
@startuml

title ***write your use case name here***     

!unquoted procedure SALT($x)
"{{
salt
%invoke_procedure("_"+$x)
}}" as $x
!endprocedure

!procedure _form()
{+
***write your code (GUI) here***
}
!endprocedure

!procedure _success()
{+
***write your code (main scenario) here***
}
!endprocedure

!procedure _error()
{+
***write your code (alternative scenario) here***
}
!endprocedure

(*) --> SALT(form)
form --> SALT(success)
form --> SALT(error)

@enduml
  2. The completed form will generate a Use Case Specification in the form of a table.
  3. The Use Case Specification table can be exported as a png file.

How to Install

There are a few steps needed to install SaltySpec, the steps consist of the following:

  1. Make sure you have Git installed on your computer. If not, install Git from the following link:
    https://git-scm.com/downloads
  2. Clone the SaltySpec repository with the following code:
    git clone https://github.com/AgileRE-2022/SaltySpec.git
  3. Open the SaltySpec folder by running this following command:
    cd SaltySpec
  4. Run the following command to install the dependencies:
    pip install -r requirements.txt
  5. Run the following command to run the application:
    python manage.py runserver
  6. Access the application through http://127.0.0.1:8000 in your browser.

Limitations

Due to our limited time constraint, this project had some limitations. These limitations are as follows:

  1. The Salt GUI can only represent a single use case.
  2. The Salt GUI is in the form of activity consisting of at least one main scenario.
  3. The Salt GUI used must implement the usage of macro.
  4. The Salt GUI Must contain Title, since it will be used as the use case title.
  5. The Salt GUI Must contain !procedure _form(), since it will be used as the part that represents GUI that is being used.
  6. The Salt GUI Must contain !procedure _success(), since it will be used as the part that represents the main scenario. It should contain what the desired output looks like.
  7. The Salt GUI Must contain !procedure _error(), since it will be used as the part that represents an alternative scenario. It should contain what the alternative output looks like.
  8. The Salt GUI Must not contain header attribute in the "!procedure _success()" and "!procedure _error()", since it will break the specification output.
  9. The Salt GUI Must contain ":" before an input field, since it will be used as the part that represents the input field.
  10. The Salt GUI Must only have one button in each interface, since it will be used as the part that represents the button.
